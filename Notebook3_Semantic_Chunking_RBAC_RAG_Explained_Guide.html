Notebook 3 — Semantic Chunking + RBAC RAG

A student-friendly guide to the SageMaker notebook that combines Bedrock embeddings, secure retrieval, RBAC filtering, and grounded answer generation.

1. What this notebook does

This notebook shows how to build a secure retrieval-augmented generation (RAG) workflow in SageMaker using Bedrock and documents stored in S3.

It combines two ideas in one pipeline:

  • Policy RAG for normal questions such as leave policy, notice period, or remote work rules.
  • RBAC-controlled salary access for salary questions, where access depends on the requester’s employee level.

Instead of using a local vector database, the notebook creates embeddings with Bedrock, keeps everything in memory, ranks content with similarity, filters it based on authorization, and then sends only the authorized context to the LLM.

Inputs used by the notebook

  • Policy documents from S3
  • A salary roster CSV from S3
  • A requester employee ID entered by the learner
  • A question entered by the learner

Outputs produced by the notebook

  • A retrieval audit table
  • An authorized context table
  • An LLM-generated RAG answer
  • A one-row request audit summary

3. Step-by-step explanation

Step 1 — Configuration

This step defines the notebook settings. It loads libraries, sets the AWS region, identifies the S3 bucket and prefixes, and specifies which Bedrock models will be used for embeddings and answer generation.

It also sets limits such as chunk size, top-k retrieval count, and how many candidate rows should be over-fetched before filtering.

Why this matters: Centralized configuration makes the notebook easier to understand and change. Learners can quickly see which parameters control the pipeline.

Step 2 — Load policy documents and salary roster from S3

This step connects to S3 and reads two types of input:

  1. Policy documents in .docx, .txt, or .md format
  2. The salary roster CSV that contains employee ID, employee name, level, role, monthly salary, and annual salary

Normal policy questions are answered from the policy documents. Salary questions use the salary roster as the controlled source of truth.

The result of this step is a clean in-memory representation of the organization’s knowledge that the next steps can chunk and index.

Step 3 — Build policy chunks and salary retrieval rows

This step prepares the content for retrieval.

  • Policy documents are split into smaller text chunks so retrieval can be more precise.
  • Salary rows are converted into short text records so each row can also be embedded and retrieved like a document chunk.

All chunks and rows are combined into a single table, often referred to as the retrieval corpus. Each item carries metadata such as document type, chunk ID, and employee ID.

What to observe: After this step, the notebook has transformed raw files into retrieval-ready units. This is the point where large documents become searchable blocks.

Step 4 — Create the retrieval matrix with Bedrock embeddings

In this step, each retrieval unit is converted into a numeric vector using a Bedrock embedding model. The notebook then stacks those vectors into an in-memory matrix.

When the learner asks a question, the question is also embedded. The notebook compares the question vector to the corpus vectors and ranks items by similarity.

Because this notebook is designed for SageMaker classroom use, retrieval happens in memory instead of through FAISS or a local vector store.

Document chunk → Embedding vector Question → Embedding vector Similarity score → Ranked candidates

Step 5 — RBAC rules and retrieval audit table

This is the security layer of the notebook.

The notebook determines whether the question is about salary. If it is, it reads the requester’s level directly from the salary DataFrame using the requester employee ID provided by the learner.

It then applies role-based access control:

  • Lower levels can typically view only their own salary
  • Higher levels can view more, depending on the rule defined in the notebook
  • For non-salary questions, salary rows are hidden completely

The retrieval audit table shows every highly ranked item and explains whether it was allowed or blocked and why.

ColumnMeaning
rankWhere the item appeared in retrieval order
doc_typeWhether the item is a policy chunk or a salary row
access_decisionWhether the item was allowed into final context
reasonWhy it was allowed or blocked
employee_idThe employee tied to that salary row, if applicable

Step 6 — LLM-generated answer over authorized context only

This step turns the notebook into a full RAG system.

The notebook creates an authorized context by selecting only the rows that survived retrieval and RBAC filtering. It then builds a prompt that tells the Bedrock model to answer only from that authorized evidence.

This means:

  • Normal policy questions are answered from policy chunks
  • Salary questions are answered only from salary rows the requester is allowed to see
  • If the needed evidence is not authorized or not present, the model should say it does not know from the authorized context

The notebook displays the exact rows sent to the model, followed by the final LLM-generated RAG answer.

Most important idea: Retrieval happens first, authorization happens second, and only then does generation happen.

Step 7 — Request audit summary

This step creates a compact one-row table that summarizes the full transaction.

It records useful audit fields such as:

  • The requester employee ID
  • The requester level that was read from the salary DataFrame, when relevant
  • Whether the question was a salary question or a normal policy question
  • How many items were retrieved
  • How many were finally authorized
  • The final answer returned by the model

This summary is helpful for debugging, governance reviews, and explaining secure AI behavior to learners.

Mini-hack — Secure retrieval challenge

The mini-hack invites learners to try several combinations of requester IDs and questions.

Good experiments include:

  • A normal policy question with no requester ID
  • A salary question where the requester asks for their own salary
  • A salary question where the requester asks for another employee’s salary
  • The same question with a different requester ID to see how authorization changes the answer

4. What learners should notice while running the notebook

Notice 1 — Retrieval is not enough

The top retrieved item may not be safe to use. Authorization decides what can actually be shown to the model.

Notice 2 — Salary and policy questions follow different paths

Policy questions search documents. Salary questions search salary rows and apply RBAC rules before generation.

Notice 3 — The answer quality depends on the authorized context

If the final context is accurate and authorized, the model can answer well. If the correct evidence is missing, the model should not guess.

5. Suggested experiments

  1. Ask a normal question such as What is the leave policy? and keep requester ID blank.
  2. Ask a salary question for your own employee ID.
  3. Ask a salary question for another employee ID and see whether the audit table blocks it.
  4. Change the requester employee ID and repeat the same question.
  5. Observe how the retrieval audit table, authorized context table, and final answer change together.
