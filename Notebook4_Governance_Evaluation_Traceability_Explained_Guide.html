Notebook 4 — Governance, Evaluation & Traceability

A student-friendly guide to the notebook that adds guardrails, audit logging, prompt comparison, and Bedrock RAG evaluation behavior.

1. What this notebook does

This notebook demonstrates how to add governance, evaluation, and traceability to a simple enterprise assistant.

It starts with a small policy assistant, then adds safety checks, audit logging, tabular review, prompt comparison, and a Bedrock-based RAG comparison between strict and loose answering behavior.

The main goal is to show that building an enterprise assistant is not only about generating answers. It is also about:

  • controlling what the assistant is allowed to answer
  • recording what happened during each request
  • reviewing behavior in tables
  • seeing how prompt design affects groundedness

3. Step-by-step explanation

Step 1 — Define a simple enterprise assistant

The notebook begins with a very small knowledge source called POLICY_STORE. This is a simple dictionary of policy names and policy text.

The assistant answers by finding relevant policy entries and returning a response. This provides a minimal baseline before any governance controls are added.

Purpose of this step: It creates the simplest possible assistant so that the later governance layers are easy to see and understand.

Step 2 — Add a pre-execution guardrail

This step introduces a guardrail using a list of blocked keywords such as salary and compensation.

If a learner asks a question that contains one of these blocked terms, the notebook stops the request before the assistant answers. This is called a pre-execution guardrail because the control happens before the answering function runs.

In real enterprise systems, guardrails can be much more advanced, but this small example clearly shows the core idea: some requests should be blocked or reviewed before the model responds.

Step 3 — Wrap the assistant with governance logging

This step records what happened during every request. The governed wrapper stores useful details such as:

  • the learner’s query
  • whether the request was allowed or blocked
  • the final response
  • latency or timing information

The notebook then runs a small test list of example queries so the log fills with several realistic entries.

Why this matters: A governed system should not be a black box. Logs make the assistant observable and reviewable.

Step 4 — Turn logs into a table

Once requests have been logged, the notebook converts the log into a pandas DataFrame.

This table is important because it turns governance from hidden code into something visible. Learners can quickly inspect:

  • which requests were blocked
  • which requests were allowed
  • what answers were returned
  • how long the requests took

When governance results are shown in a table, it becomes much easier to debug behavior and explain system decisions.

Step 5 — Track prompt and system drift

This step defines two different system prompt styles:

  • Strict — answer only from the provided policy context
  • Loose — use the policy context first, but allow broader reasoning or inference if needed

The purpose of this comparison is to show that prompt wording changes the behavior of the assistant. Even when the same model is used, a strict system prompt can lead to grounded answers, while a looser prompt can lead to inferred answers that are not fully supported by evidence.

Main lesson: Governance is not only about blocking harmful requests. It is also about controlling how the model reasons and how much freedom it has beyond the retrieved context.

Step 6 — Compare strict vs loose RAG behavior with Bedrock

This is the main Bedrock RAG step of the notebook.

The notebook first retrieves a small set of relevant policy rows for a given question. It then builds a context block and sends the same retrieved evidence to Bedrock twice:

  1. once with the strict prompt
  2. once with the loose prompt

The notebook uses two types of questions:

  • A known query — the answer is clearly present in the policy store
  • An unknown query — the answer is not fully present in the retrieved context

This creates a useful contrast:

  • For the known query, both modes may look similar because the evidence is clear.
  • For the unknown query, the strict mode should say it does not know from the retrieved policy context, while the loose mode may produce an inference.

The final comparison table helps learners see how groundedness changes when the prompt becomes stricter or looser.

Mini-hack — Build one governance improvement

The notebook ends with a small extension activity. Learners are asked to add one more governance feature.

Examples include:

  • adding a human-review topic list
  • adding a new blocked keyword category
  • marking certain queries for escalation rather than blocking them completely
  • adding a confidence label or governance status column to the audit table

Wrap-up

The wrap-up section reinforces the central message of the notebook: a useful enterprise assistant needs more than answer generation. It needs controls, logs, comparison views, and evidence-based answering behavior.

4. What learners should notice while running the notebook

Notice 1 — A simple assistant is not automatically safe

Before guardrails are added, the assistant may try to answer anything. Governance adds policy boundaries.

Notice 2 — Tables make governance visible

The audit DataFrame shows exactly what happened. This is much easier to inspect than raw print output.

Notice 3 — Prompt wording changes groundedness

Strict prompts encourage evidence-based answers. Loose prompts allow more inference and therefore more risk.

5. Suggested experiments

  1. Add a new blocked topic and test whether the guardrail catches it.
  2. Change the known query and watch how retrieval changes.
  3. Change the unknown query and compare the strict and loose answers again.
  4. Add a new policy line to the policy store and see whether retrieval and answer quality improve.
  5. Add a new audit column to the log DataFrame, such as review status or business risk.
