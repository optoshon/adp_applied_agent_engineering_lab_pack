Notebook 5 — Multimodal RAG on AWS Bedrock

A student-friendly guide to the SageMaker notebook that combines Docling extraction, Bedrock multimodal understanding, embeddings, retrieval, and final grounded answer generation.

1. What this notebook does

This notebook teaches a full multimodal RAG pipeline on AWS.

The notebook reads source files from S3, extracts multiple modalities with Docling, converts each modality into retrieval-ready content, and finally sends the best multimodal evidence to a Bedrock model for grounded answer generation.

The three modalities used are:

  • Text extracted from documents
  • Tables extracted as structured tabular data
  • Images extracted from figures and then converted into structured JSON with a Bedrock vision-language model

3. Step-by-step explanation

Step 0 — Install dependencies

This step installs the packages needed for the notebook. The important categories are Docling for extraction, boto3 for AWS access, and pandas/numpy/Pillow/python-docx for handling notebook data.

Step 1 — Imports and notebook utilities

This step imports the Python packages used across the notebook. The notebook uses DataFrames heavily so that each stage stays visible and easy to understand.

Step 2 — Configuration

This step sets the AWS region, S3 location, model IDs, chunk sizes, and temporary workspace. It is the only cell learners usually need to edit before running the notebook.

Step 3 — Bedrock model discovery

This step lists the Bedrock models available in the current region and tries to select one embedding model, one image-capable model, and one answer-generation model.

Main idea: model availability varies by region and account, so discovery is useful before building the rest of the pipeline.

Step 4 — S3 download and temporary workspace

This step reads the source file list from S3 and downloads the files into a temporary notebook workspace. The notebook uses S3 only for input and does not write outputs back to S3.

Step 5 — Docling extraction

This is the multimodal extraction step. For each source document, Docling extracts text blocks, tables, and images. The notebook stores them in DataFrames so learners can inspect each modality separately.

Step 6 — Text chunking and table chunking

Text and tables are prepared differently. Text is split into overlapping chunks. Tables are split by rows with overlap while preserving headers and table identity. This creates retrieval-ready content for both modalities.

Step 7 — Image understanding with Bedrock

Each extracted image is sent to a Bedrock multimodal model. The model returns strict JSON that describes the image. This JSON becomes the searchable representation of the image for later retrieval.

Why this matters: the notebook retrieves semantic descriptions of images, not raw image pixels.

Step 8 — Build one multimodal corpus

This step merges text chunks, table chunks, and image-description chunks into one combined corpus. Each row carries a stable chunk ID, modality label, source key, and searchable content.

Step 9 — Bedrock embeddings and in-memory retrieval

Every row in the multimodal corpus is converted into an embedding vector using a Bedrock embedding model. The notebook keeps those embeddings in memory and uses cosine similarity for retrieval instead of a local vector database.

Step 10 — Multimodal retrieval and context building

The notebook retrieves the best evidence for the learner’s question. It picks top chunks per modality, merges them, applies a lightweight rerank score, and creates a final context block with indices such as [1], [2], and [3].

Step 11 — Final multimodal RAG answer

The learner’s question and the final multimodal context are sent to a Bedrock text model. The prompt instructs the model to answer only from the provided evidence and to cite supporting blocks.

Step 12 — Observability tables

The notebook ends by showing run summary, extraction manifest, and retrieval ranking tables. These make the pipeline easier to explain and easier to debug.

Mini-hack

Ask different questions and observe whether the final answer relies more on text, tables, or image descriptions. Then change retrieval settings and compare the results.

4. What learners should notice while running the notebook

Notice 1 — Multimodal RAG is more than OCR

The notebook does not just read images as text. It creates structured semantic descriptions of images and retrieves those descriptions alongside text and tables.

Notice 2 — Different modalities need different preparation

Text is chunked by content length, tables are chunked by rows, and images are converted into JSON first. That is why multimodal RAG usually has separate modality stages.

Notice 3 — The final answer is grounded in a mixed context

The final context can contain evidence from text, tables, and image descriptions at the same time. That is the main advantage of multimodal retrieval.

5. Suggested questions for practice

  • What policy details are described in the documents?
  • What values or limits are shown in the extracted tables?
  • Does any figure or diagram add detail that is not obvious from text alone?
  • Which modality contributed the most useful context for this answer?
